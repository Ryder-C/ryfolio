I am currently working on [Ryguessr](/projects/ryguessr) (a geoguessr clone) and encountered the problem of choosing random
colors for players. The naive approaches came with compromises:

- **Picking a random hex color** led to players sometimes getting near-identical colors which made the map markers
  hard to tell apart.
- **Rotating through a pre-defined list of "nice" colors** works, but I didn't want to restrict the number of players or
  give duplicate colors.

I wanted something that:

- **(a)** produces a stream of colors that are _actually_ perceptually distinct.
- **(b)** adapts when players change their color or leave.
- **(c)** has very low computational cost.

## Working in a perceptual color space

Humans don't see distances in color the same way that monitors emit it. Some two pairs of sRGB[^1] colors can be the same
Euclidean distance apart, and yet one pair may look nearly identical while the other looks completely different.

<SrgbDistanceDemo client:load />

The fix is to instead use a **perceptually uniform color space**, where Euclidean distance accurately reflects perceived difference.

<Callout title="Perceptually Uniform Color Space" variant="definition">
  A color space where the Euclidean distance between two points is proportional
  to how different those colors look to the human eye.
</Callout>

I settled on [**OKLAB**](https://en.wikipedia.org/wiki/Oklab_color_space)[^2] for this project, a 3D color space (like sRGB) with axes labled: $L$, $a$, and $b$.

You can clearly see the difference between these color spaces by interpolating different pairs of colors. Equal steps in sRGB produce very different perceptual steps, while equal steps in OKLAB _look_ equal as well.

<Callout variant="remark" defaultOpen={false}>
  Part of the sRGB gradient's *muddiness* is [gamma
  encoding](https://en.wikipedia.org/wiki/Gamma_correction) rather than
  perceptual non-uniformity, but since sRGB is what monitors actually display,
  it's a fair comparison. OKLAB addresses this as well.
</Callout>

### OKLAB &rarr; linear sRGB

In order to display OKLAB colors, they must be converted to sRGB since that is what displays use. {/* Look into what color spaces monitors support for display */}

OKLAB is defined as a transformation on top of [LMS](https://en.wikipedia.org/wiki/LMS_color_space) (a model of how the human eye responds to light), which is itself a transformation on top of linear sRGB. So converting OKLAB to linear sRGB is just unwinding that chain.

{/* prettier-ignore */}
<Callout variant="notation">
    - $l$ is perceptual luminosity, or brightness (0 = black, 1 = white).
    - $a$ is the green–red axis (negative green, positive red).
    - $b$ is the blue–yellow axis (negative blue, positive yellow).
</Callout>

{/* Add docstring here and change original code */}

```rust title="server/src/colors.rs"
fn oklab_to_linear_srgb([l, a, b]: [f64; 3]) -> [f64; 3] {
    // OKLAB -> LMS

    // conversion matrix
    let l_ = l + 0.3963377774 * a + 0.2158037573 * b;
    let m_ = l - 0.1055613458 * a - 0.0638541728 * b;
    let s_ = l - 0.0894841775 * a - 1.2914855480 * b;

    // cube
    let l = l_.powi(3);
    let m = m_.powi(3);
    let s = s_.powi(3);

    // LMS -> linear sRGB
    [
        4.0767416621 * l - 3.3077115913 * m + 0.2309699292 * s,
        -1.2684380046 * l + 2.6097574011 * m - 0.3413193965 * s,
        -0.0041960863 * l - 0.7034186147 * m + 1.7076147010 * s,
    ]
}
```

The matrix constants and full derivation are in [Björn Ottosson's original OKLAB post](https://bottosson.github.io/posts/oklab/).

### Linear sRGB &rarr; sRGB (gamma encoding)

Before these colors can be displayed, **gamma encoding** must be applied. The sRGB values received by displays are not _linear_ measures of light intensity. Doubling an sRGB value (say $64 \rightarrow 128$), doesn't result in twice as much light being emitted.

<Callout variant="intuition">
  Human eyes can perceive differences in dark shades better than bright ones. If
  brightness was stored linearly in 8 bits, many of the 256 values would be
  wasted on bright shades that are not visually distinguishable, while adjacent
  dark shades are significantly different. _Recall [this
  section](#working-in-a-perceptual-color-space)_.
</Callout>

This is why the linear sRGB output cannot simply be scaled by 255. sRGB's transfer function must be applied first (this is called [gamma correction](https://en.wikipedia.org/wiki/Gamma_correction)):

$$
c = \begin{cases} 12.92 \, c_{\text{lin}} & c_{\text{lin}} \le 0.0031308 \\[4pt] 1.055 \, c_{\text{lin}}^{1/2.4} - 0.055 & c_{\text{lin}} > 0.0031308 \end{cases}
$$

In code:

```rust title="server/src/colors.rs"
fn linear_to_srgb(c: f64) -> f64 {
    let c = c.clamp(0.0, 1.0); // clamp to remain in sRGB
    if c <= 0.0031308 {
        12.92 * c
    } else {
        1.055 * c.powf(1.0 / 2.4) - 0.055
    }
}
```

There exists colors in OKLAB which correspond to no color in sRGB, as they exceed the bounds of the color space. Rather than making the color space transformation fallible, the transformed colors may simply be clamped to the sRGB color space. This, naturally, distorts the projection, so colors that project out-of-bounds will shift in hue slightly. In practice, this isn't a problem.

<Callout variant="note">
  The proper way to do this is explained by Björn Ottosson (the creator of
  OKLAB) in [his post on gamut
  clipping](https://bottosson.github.io/posts/gamutclipping/).
</Callout>

## Sampling OKLAB: the $R_3$ quasirandom sequence

Now that we have a nice perceptual color space that can be converted to sRGB, the next question is: **How do we sample this space to produce maximally distinct colors?** To reword the problem formally, we need to generate an unbounded sequence of points $\{p_1,p_2,p_3,...\}$ such that each new point $p_{k+1}$ maximizes its minimum distance to all previous points $\{p_1,...,p_k\}$.

This is known as a [low-discrepancy sequence](https://en.wikipedia.org/wiki/Low-discrepancy_sequence) (also called a quasirandom sequence).

<Callout variant="definition" title="Discrepancy">
  Discrepancy is a measure of how uniformly a sequence fills a space. A
  low-discrepancy sequence fills the space more uniformly than random sampling.
</Callout>

<Callout variant="note">
  There are several well known low-discrepancy sequences (Halton, Sobol,
  Hammersley), but I went with the $R_d$ family (popularized by [Martin
  Roberts](https://extremelearning.com.au/unreasonable-effectiveness-of-quasirandom-sequences/)).
  It has no base or direction numbers to pick, the recurrence is a single
  multiply and mod, and it performs comparably to Halton for our purposes. It's
  also the most elegant in my opinion, which matters when you're doing something
  unnecessary anyway.
</Callout>
{/*greatly simplify note here*/}

Since OKLAB is three dimensional, the low-discrepancy sequence (sampled colors) should be as well. The simplest option is to run independent one-dimensional sequences on each axis and hope it works out. It doesn't, but walking through _why_ is a nice way to demonstrate the motivation for the final solution.

### Starting in 1D: the golden ratio

{/* explain why we are using Rd sequences */}

Given the golden ratio $\varphi = \frac{1+\sqrt{5}}{2}$, the $R_1$ sequence is

$$
    x_n = \frac{n}{\varphi} + x_0 \mod 1
$$

where $x_0$ is a seed offset. This sequence works beautifully in one dimension:

### Generalizing to higher dimensions

The fix is to generalize the golden ratio to higher dimensions. In 2D we use the [plastic ratio](https://en.wikipedia.org/wiki/Plastic_ratio) $\rho \approx 1.3247$, the real root of $x^3 = x + 1$. Just as $\varphi$ is the most irrational number in 1D (hardest to approximate with rationals), the plastic ratio's powers $\frac{1}{\rho}$ and $\frac{1}{\rho^2}$ are the 2D equivalent (the pair hardest to approximate jointly by rationals).

For 3D we step up one more dimension. We want the real root of $x^4 = x + 1$, which I call $\varphi_3$ ($\approx 1.2207$). The offsets are its first three powers:

$$
    \alpha_1 = \frac{1}{\varphi_3}, \quad \alpha_2 = \frac{1}{\varphi_3^2}, \quad \alpha_3 = \frac{1}{\varphi_3^3}
$$

And the $R_3$ sequence is

$$
    \mathbf{p}_n = \left(n \cdot \alpha_1 + x_0,\; n \cdot \alpha_2 + y_0,\; n \cdot \alpha_3 + z_0\right) \mod 1
$$

Each new point tends to land in a large gap left by the previous points, because the irrational step sizes never align into a repeating pattern along any axis or diagonal.

### Mapping $R_3$ into OKLAB

$R_3$ generates points in the unit cube $[0,1]^3$, but OKLAB is not entirely inside that cube. $L \in [0,1]$, but $a,b$ are roughly bounded by $\pm 0.4$ with only a small slice of that producing colors that are in the sRGB space. We linearly map each $R_3$ coordinate into a chosen range:

```rust title="server/src/colors.rs"
const L_RANGE: (f64, f64) = (0.35, 0.85);
const A_RANGE: (f64, f64) = (-0.15, 0.15);
const B_RANGE: (f64, f64) = (-0.15, 0.15);
```

Two things are happening here.

1. **Lightness stays in the middle.** Values below $0.35$ are too muddy and dark and above $0.85$ they look washed out. So we clamp between those values.

2. **Chroma stays low enough.** The $\pm 0.15$ bounds on $a$ and $b$ keep us mostly inside the sRGB bounds so the clamping mentioned in the [previous section](#linear-srgb--srgb-gamma-encoding) is minimally encountered. {/* minimalyl encountered */}

## Handling players joining and leaving

Generating the $n$-th color is easy. The harder part is staying perceptually distinct as players come and go, without causing existing player colors to change.

The approach I took is each player holds onto their index for the lifetime of their session. When a player leaves, their index goes back into a pool of returnable indices. When a new player joins, attempt to reuse a returned index before advancing to a new one.

```rust title="server/src/colors.rs"
pub struct DistinctColors {
    seed: [f64; 3],
    next_index: usize,
    returned_indices: BTreeSet<usize>,
}
```

But players can also pick their own custom colors and we don't want the server's generated colors to collide with those either. So when we try to get a color for a player, we don't just grab the immediate next index, but we check that its color is significantly perceptually distinct from every color currently in the lobby.

So our `next_filtered` function runs through two parts:

1. **Try returned indices first.** For each returned index, check whether its color is distinct from the currently occupied colors. If yes, use it. If not, put it back in the pool and keep looking.
2. **Fall back to fresh indices.** If no returned index works, advance `next_index`. If that color happens to be too close to an already occupied one (someone's custom color), then put that index in `returned_indices` and get the next one.

This way even a fresh index that gets blocked by a custom color isn't wasted, it's just waiting to be used from the pool of returned indices.

There's one issue. If a returned index doesn't work on this allocation, we put it back and it stays in the pool. This means the pool can grow if the currently-occupied colors keep blocking the same returned indices. In practice the pool size is bounded by the number of players who have ever been in the game, which for Ryguessr is small enough not to matter.

<Callout variant="note">
  The filter only guards generated colors against occupied ones. For example,
  two players who both pick <span style={{ color: '#ff0000' }}>#ff0000</span> as
  a custom color will still collide. Preventing this goes against my idea of not
  wanting to change existing colors so I just let it be.
</Callout>

### How close is too close?

Two colors count as "distinct" if their Euclidean distance in OKLAB is at least 0.04. Where does that number come from?

0.01 to 0.02 is often cited as the just noticeable difference in OKLAB. So you would have a hard time telling the difference between a pair of colors with that distance. For example <span style={{ color: '#4a90e2'}}>#4a90e2</span> and <span style={{ color: '#5e8ce1'}}>#5e8ce1</span> have a distance of approximately 0.02. Choosing 0.04 gives a comfortable margin where colors remain clearly distinguishable.

```rust title="server/src/colors.rs"
const THRESHOLD_SQ: f64 = 0.04 * 0.04;
```
